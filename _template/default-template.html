모두의충전 — 고객 세그먼트 분리 & 타임특가 설계
2024.05 ~ 2024.06
EV충전 인프라에서 발생하는 실시간 산업 데이터와 결제 데이터를 함께 해석해 고객 세그먼트를 처음으로 분리한 프로젝트입니다. 핵심 타겟인 도심 직업 운전자에게 최적화된 타임특가를 설계해 충전 결제 매출 60억, 전년 대비 +20% 성장을 이끌었습니다.
01 · PROBLEM

앱 가입자는 있었지만 누가 매출을 만드는지 알 수 없었다. 성별·연령 같은 인구통계 정보만 있을 뿐, 실제 충전 행동으로 고객을 분류한 적이 없었고 — 어떤 고객이 왜 다시 오는지 파악 자체가 불가능했습니다.

PAIN POINT · 도심 택시·배달 운전자

충전 대기 시간은 곧 수입 공백이라 빠른 충전이 최우선인데, 요금이 비싸면 경쟁 사업자 기기로 이동해야 한다. 저렴하고 빠른 충전이 가능한 시간과 위치를 알 방법이 없어 매번 비효율이 반복된다.

PAIN POINT · 스칼라 서비스팀

어떤 고객군이 실제 매출을 만드는지 행동 데이터가 없어 타겟 마케팅이 불가능했다. 충전 시간대별·위치별 수요 패턴을 몰라 요금 정책을 일괄 적용할 수밖에 없는 상태였습니다.

02 · ANALYSIS · 고객 세그먼트 분석
버블 크기 = 고객 수 비중 · 라이트·일반 유저는 많지만 충성도가 낮아 매출 기여가 낮았습니다.
03 · FINDING · 핵심 타겟 발견
핵심 타겟 — 도심 택시·배달·택배 운전자

하루에 여러 번 충전이 필요한 직업 운전자. 충전 대기 시간은 곧 수입 공백이라 빠른 충전·낮은 요금이 최우선이고, 도심 급속충전기를 집중적으로 사용해 결제 빈도와 단가 민감도가 모두 높았습니다.

타임특가 설계 기준

도심 급속충전기 한정 적용. 새벽·오전·오후·퇴근 시간대 — 핵심 타겟이 자주 충전하는 시간대에 집중적으로 낮은 단가를 제공해, 경쟁 사업자 기기 대신 모두의충전으로 재방문하도록 유도했습니다.

PAIN POINT · 충전 유저

포인트를 쌓아도 쓸 타이밍을 모릅니다. KT 포인트는 1,000원 단위 쿠폰 방식이라 번거롭고, 블루멤버스 포인트는 모두의충전에서 쓸 방법이 없었습니다. 결국 포인트는 쌓이지만 재결제로 이어지지 않습니다.

PAIN POINT · 스칼라 서비스팀

미사용 포인트가 수억 원 규모로 누적되어 재무적으로 부채로 계상되는 구조. 포인트를 사용하지 않는 비활성 유저가 늘어날수록 재방문율이 낮아지는 악순환이 지속됐습니다.

모바일 상품권 성공 이후, 오프라인에서도 동일한 선물 경험을 제공하고 싶었습니다. 기존 실물 상품권은 직접 전달해야만 했지만, 직접 전달하지 않아도 브랜드 경험이 살아있는 선물을 만들고 싶었습니다. 단, 카드 제작비 + 투봉 인편배송 비용이 발생하는 상황에서도 고객 경험을 양보할 수 없었습니다.

PAIN · 받는 사람

기존 실물 상품권은 직접 전달해야 해서 선물하기 번거로움. 받는 사람 입장에서도 앱 잔액과 따로 관리되면 불편하고 특별함이 없다.

CONSTRAINT · 비용 구조

카드 제작비 + 투봉 포장 + 인편배송 비용이 고정 발생. 이 비용을 부담하면서도 수익이 나고, 고객 경험도 살아있어야 했습니다.

코나아이 — 전사 백오피스 통합 재설계
기존 백오피스에 새 기능을 추가하는 데 시간이 너무 오래 걸리는 구조적 문제를 해결한 리뉴얼 프로젝트. 전사 이해관계자 인터뷰로 Pain Point를 파악하고 IA를 재수립해, 기능 추가 속도와 운영 효율을 동시에 개선했습니다.
01 · PROBLEM

백오피스에 기능을 추가하는 데 너무 오랜 시간이 걸렸습니다. 비즈니스가 빠르게 늘어나면서 새 기능 요청이 쌓였지만, 기존 구조로는 빠르게 대응할 수가 없었습니다. 개발 대기가 길어질수록 운영팀의 수작업도 늘어나는 악순환이 반복됐습니다.

INTERNAL TEAM · 운영·정산팀

데이터 추출이 필요할 때마다 개발팀에 요청 → 개발팀도, 운영팀도 비효율. 정산·통계·배송 기능이 파편화되어 부서 간 연계 작업 시 수작업이 반복됩니다.

DEVELOPMENT · 기능 추가 병목

새 기능 요청마다 구조 파악부터 다시 해야 해서 개발 리드타임이 길어집니다. 운영 DB에 직접 접근하는 관행으로 성능 이슈도 반복 발생했습니다.

02 · ANALYSIS · 전사 이해관계자 인터뷰
IA 재수립 전, 부서별 실제 Pain Point를 직접 인터뷰로 파악했습니다.
인터뷰 대상 부서
  • → 운영팀 (지자체 관리·CS)
  • → 정산팀 (매입사 정산)
  • → 마케팅팀 (통계·리포트)
  • → 개발팀 (데이터 추출 요청 수신)
공통 Pain Point 발견
  • → "데이터 뽑을 때마다 개발팀 대기"
  • → "정산 항목이 분산되어 통합 확인 불가"
  • → "신규 지자체 추가 시 2~3주 소요"
  • → "통계 그래프가 없어 직접 엑셀 작업"
03 · SOLUTION · 3가지 재설계
①
IA 전면 재수립 — 5개 파트로 구조화

파편화된 기능을 회원·결제·상품·정산통계·서비스 5개 파트로 재분류. 각 파트가 독립적으로 운영되면서 서로 연계되는 구조를 설계했습니다.

②
시계열 통계 시각화 — 개발팀 의존 제거

일·주·월·년 단위 시계열 통계를 어드민에서 직접 조회·다운로드 가능하도록 구현. 매번 개발팀에 요청하던 데이터 추출을 운영팀이 독립적으로 처리합니다.

③
다중 매입사 정산 통합 UI

여러 매입사(카드사·지자체)별로 분산된 정산 정보를 하나의 화면에서 조회·처리 가능하도록 통합. 정산팀의 월별 결산 작업 시간을 크게 단축했습니다.

04 · RESULT
개발팀 데이터 요청 대폭 절감

운영팀 셀프 데이터 조회로 개발팀 리소스 확보. 부서 간 대기 시간 제거.

신규 비즈니스 확장 인프라

새 지자체·새 상품 추가 시 어드민 설정만으로 처리. 개발 공수 대폭 절감.

부서 간 협업 효율 극대화 개발팀 데이터 추출 리소스 대폭 절감 IA 5개 파트로 전면 재구조화 신규 비즈니스 확장 핵심 인프라
KEY INSIGHT · 배운 점

많은 기능 개선보다 확장성에 초점을 두고 만들어야 한다. 기술 부채는 일찍 해결할수록 좋다. 내부 시스템 설계도 결국 사용자 인터뷰에서 시작해야 진짜 Pain Point를 잡을 수 있다.

ANALYSIS

미리디 서비스 분석

데이터가 보여주는 구조적 문제 4가지

ANALYSIS 01
가입 고객층 ≠ 결제 고객층

가입자 절반이 학생인데 결제는 소상공인이 합니다. 이 간극이 0.6% 전환율의 본질입니다.

가입자 구성
학생 46% 직장인 34% 기타 20%
결제 고객 구성
소상공인 72% 기타 28%
왕관 뱃지 하나로 46% 학생 + 34% 직장인을 전환하려는 구조 → 전환율이 낮은 건 결과가 아니라 예고된 수치.
왕관 뱃지에 반응하는 고객은 "빠르게 만들고 싶은" 고객이 아니라, "더 아름답게 만들고 싶은" 고객입니다. 세그먼트가 다르면 트리거도 달라야 합니다.
ANALYSIS 02
가장 많이 쓰는 기능은 PPT 제작
많은 유저에게 미리캔버스는 "무료 PPT 툴"로 인식됩니다.
